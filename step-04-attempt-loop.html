어템프트 루프 · while (true)

한 턴이 실제로 조립·실행되는 심장부. 아웃터 재시도/페일오버 드라이버가 이너 어템프트를 반복 디스패치하고, 매 결과를 분류해 살아남는다.

무엇을 하나 (반복 1회)

  • retry-limit 가드 — 한도 초과면 handleRetryLimitExhaustion으로 강제 종료(livenessState "blocked").
  • 프롬프트 + 런타임 플랜 빌드 — 이번 반복의 입력을 구성.
  • 이너 어템프트 1회 디스패치 — runEmbeddedAttemptWithBackend → backend.ts → harness/selection → runEmbeddedAttempt(attempt.ts:837). 여기 안에서 조립 10단계 → 디스패치 → stop-reason이 일어난다.
  • 결과 normalize + 분류 — 결과를 정규화한 뒤 continue(재시도/프로필 회전/모델 폴백/컴팩션) 또는 terminal 반환.

코드 위치 (검증됨)

  • run.ts:1885 while (true) 어템프트 루프
  • run.ts:1886–1920 retry-limit 가드 + handleRetryLimitExhaustion
  • run.ts:2044 이너 어템프트 디스패치 runEmbeddedAttemptWithBackend
  • backend.ts:10 → harness/selection.ts:325 → attempt.ts:837 runEmbeddedAttempt
  • run.ts:2235 normalizeEmbeddedRunAttemptResult (분류 입력)

"조립 10단계"란

  • 이너 어템프트가 prepStages.mark()로 자기 단계를 계측한다 — 그 마커 순서가 곧 조립 파이프라인이다(attempt.ts:955 → 2943).
  • 5번째(code-mode/tool-search)는 조건부라, 일반 실행에선 보통 9개가 발화한다. 10은 최대치.
  • 이 단계들은 모델 호출 直前까지의 조립만 덮는다. 실제 모델 호출·도구 루프·stop-reason은 그 뒤(디스패치)에서 일어난다.
