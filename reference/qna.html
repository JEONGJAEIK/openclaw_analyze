Q1 에이전트 런타임에 "embedded 에이전트"가 따로 있어?
한 줄 답 별개의 "또 다른 에이전트"가 아니라, 에이전트를 돌리는 방식이다. "embedded"는 OpenClaw가 자기 프로세스 안에서 직접 모델 루프를 구동한다는 뜻 — 그 전용 런타임이 src/agents/embedded-agent-runner/이고 진입점은 runEmbeddedAgent()다.

핵심: "embedded vs CLI"는 모델 루프를 누가 소유하느냐의 차이

런타임에는 두 갈래가 있다. 같은 에이전트라도 어느 갈래로 실행되느냐만 다르다.

Embedded harness · 내장
OpenClaw가 직접 모델 루프를 돈다

OpenClaw가 준비한 에이전트 루프 안(in-process)에서 모델을 호출한다. 재시도·페일오버·컴팩션 같은 운영 껍데기를 OpenClaw가 입힌다.

기본 openclaw 런타임 + 플러그인 하네스 codex · copilot
CLI backend · 외부
별도 CLI 프로세스가 루프를 돈다

로컬 CLI 바이너리를 별도 프로세스로 띄워 그 프로세스가 모델 루프를 소유한다. OpenClaw는 model ref만 표준으로 유지한 채 위임한다.

예: claude-cli — "embedded harness가 아님"이라 문서에 명시

그래서 "따로 있냐"는 두 갈래로 답한다

  • 또 다른 에이전트 인스턴스로 따로 있나? → 아니다. embedded는 별개 에이전트가 아니라 실행 방식이며, 기본값이다.
  • 전용 코드/런타임으로 구분되나? → 그렇다. embedded-agent-runner라는 본체가 있고 외부 CLI 백엔드와 명확히 갈린다.

헷갈리기 쉬운 동음이의 — 이건 다른 개념

src/infra/embedded-mode.ts의 "embedded mode"는 OpenClaw가 데스크탑/모바일 호스트 앱 안에 내장되어 도느냐의 플래그로, 에이전트 런타임과는 무관하다.

Q2 에이전트는 몇 대야? 한 대에 여러 모델이 붙어?
한 줄 답 고정 1대가 아니다. 설정된 상위 에이전트(기본 main 1대) 위에, 실행 중 동적으로 늘어나는 sub-agent 트리가 얹힌다. 그리고 한 에이전트에는 여러 모델이 붙는다 — primary+fallback 체인에 역할별 보조 모델까지.

A. 몇 대인가 — 두 층으로 본다

1
설정 에이전트(페르소나) — 기본 main 1대
각자 고유한 workspace · agentDir · 인증 프로필 · 모델 레지스트리 · 세션 스토어를 ~/.openclaw/agents/<id>/에 가진 완전한 인격. openclaw agents add <id>로 여러 대를 나란히 둘 수 있고, bindings로 라우팅한다.
2
Sub-agent — 실행 중 동적으로 fan-out
한 번의 실행에서 백그라운드 sub-agent를 sessions_spawn으로 띄운다. 개수는 인메모리 레지스트리로 카운트되며 고정값이 없다. 상한: maxSpawnDepth 1(1–5) · maxChildrenPerAgent 5 · maxConcurrent 8.

B. 한 대에 여러 모델 — 그렇다, 네 가지 방식으로

  • ① primary + fallback 체인 — 모델 필드 자체가 { primary, fallbacks[] }. primary 실패 시 순서대로 다음 모델로 넘어간다.
  • ② 모델 allowlist / alias 카탈로그 — 한 에이전트가 여러 provider/model을 alias·params·runtime으로 매핑해 들고 있다.
  • ③ 역할별 보조 모델 — 작업 종류마다 다른 모델을 붙인다(아래 그리드).
  • ④ 런타임 스위칭 + 인증 로테이션 — /model 실시간 전환, provider 안에서 여러 auth 프로필을 쿨다운으로 돌린다.

Q3 대화 내용은 어디에 저장돼?
한 줄 답 세션마다 JSONL 파일 한 개로 디스크에 쌓인다 — ~/.openclaw/agents/<agentId>/sessions/<sessionId>.jsonl (기본 에이전트 main). 한 줄이 곧 한 항목이며 append-only다.

한 줄 = 한 JSON 항목

첫 줄은 세션 헤더(type:"session" · version 3 · cwd), 이후로는 전부 type:"message" 행이다. user / assistant / toolResult 메시지가 줄 단위로 쌓인다.

  • 도구 호출은 assistant 메시지의 content 배열 안에, 그 결과는 별도의 toolResult 메시지로 저장된다.
  • thinking·이미지(base64)·model_change·compaction 같은 항목도 같은 파일에 줄로 들어간다.
  • 작성자는 SessionManager.persistRecord → append 헬퍼(transcript-jsonl.ts). 파일 권한은 0600.
# 메시징 런타임(기본) ~/.openclaw/agents/main/sessions/<sessionId>.jsonl # 독립 에이전트 하네스(단수 agent 폴더) ~/.openclaw/agent/sessions/<timestamp>_<sessionId>.jsonl

포맷: JSONL — 일반 JSON도 SQLite도 아니다. 저장 루트는 ~/.openclaw(OPENCLAW_STATE_DIR로 변경 가능).

Q4 세션 정보는 어디에 저장돼?
한 줄 답 에이전트마다 JSON 파일 한 개에 모인다 — ~/.openclaw/agents/<agentId>/sessions/sessions.json. sessionKey → 세션 메타 맵으로, 대화 본문이 아니라 메타데이터와 포인터를 담는다.

대화 파일과 같은 폴더, 다른 역할

한 에이전트의 모든 세션이 이 단일 JSON 객체 하나에 들어간다. 세션마다 메타 파일이 따로 있는 게 아니다. 각 항목의 sessionFile 필드가 짝이 되는 .jsonl 대화 파일을 가리킨다.

  • 담는 것: sessionId · sessionFile 포인터 · 타임스탬프 · status · 모델/런타임 선택(override) · 작업 디렉토리 · 큐 상태(queueMode 등) · 제목/origin(채널·발신자) · resume 정보 · 토큰·컴팩션 통계.
  • 작성자: store.ts의 saveSessionStore / updateSessionStore — temp 파일 + rename으로 atomic 저장, 경로별 writer 락으로 직렬화.
  • "지금 보던 세션" 포인터는 별도 파일: ~/.openclaw/tui/last-session.json (TUI 레이어).
  • 모든 세션 상태의 소유자는 gateway다. UI는 파일을 직접 읽지 않고 gateway에 질의한다.

포맷: 들여쓴 단일 JSON 객체 — JSONL이 아니다(대화는 JSONL, 메타는 JSON).

한눈에 — 대화(Q3)와 세션 정보(Q4)는 같은 폴더에 나란히 산다

메타(sessions.json)가 색인이라면, 대화 본문(<id>.jsonl)은 그 색인이 가리키는 실제 기록이다.

~/.openclaw/agents/main/sessions/ ├─ sessions.json ← 세션 정보(메타) · Q4 — sessionKey→메타 맵 (JSON) ├─ <sessionId-A>.jsonl ← 대화 내용 · Q3 — 한 줄 = 한 메시지 (JSONL) ├─ <sessionId-B>.jsonl ← 대화 내용 · Q3 └─ <sessionId-C>.jsonl ← 대화 내용 · Q3
sessions.json — 한 에이전트의 모든 세션 메타를 담은 단일 JSON 맵. 각 항목의 sessionFile이 아래 대화 파일을 가리킨다.
<sessionId>.jsonl — 세션 1개당 1파일. append-only JSONL 트랜스크립트로 실제 user/assistant/tool 메시지가 쌓인다.
