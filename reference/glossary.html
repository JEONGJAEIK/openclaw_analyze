Reference · 용어 사전

용어 사전 — 개념을 따라 건너다니기

런타임 사다리에 흩어진 핵심 용어를 한자리에 모았다. 각 용어는 그 개념이 가장 깊이 설명되는 home 페이지로 이어지고, 아래 등장 줄은 그 용어가 본문 링크로 걸린 다른 단계들 — 즉 어디서 이 개념을 마주치는지를 되짚는 역링크다. 순서대로 읽지 않고 개념을 따라 옮겨 다닐 수 있게 한, 런타임 사다리 묶음의 나무위키식 색인. 코랄색 용어는 메커니즘을 더 파고드는 심화 페이지다.

A · 진입

진입 & 설정 해석 (문지기) step-01-entry

모든 출처(채팅·예약 작업·명령·다른 에이전트)의 요청이 모이는 단일 공개 진입점. 요청을 정돈하고, 기본값을 채우고, 생애 도장을 찍는 얇은 한 겹.

등장 ← 설정 스냅샷 · 생애 도장

설정 스냅샷 심화 step-01-config-snapshot

켜질 때 한 번 읽어 프로세스 메모리에 박아 두고 모두가 빌려 쓰는, 해석이 끝난 설정 사본. 비밀값이 풀린 상태로 갈무리되고, 바뀌면 통째로 교체되며 판 번호로 캐시를 자동 무효화한다.

등장 ← 진입(A) · 생애 도장 · 세션 레인

생애 도장 심화 step-01-lifecycle-stamp

실행이 서버의 몇 번째 생애에 속하는지 찍는, 프로세스에 하나뿐인 세대 표식(UUID). 인-프로세스 재시작 때 새 UUID로 회전해, 옛 실행을 유령으로 솎아낸다.

등장 ← 진입(A) · 설정 스냅샷 · 세션 레인

B · 세션 · 큐

세션 해석 + 2단 레인 큐 step-02-session-queue

요청이 어느 대화 소속인지 하나의 이름표(대화 주소)로 확정하고, 세션 레인(같은 대화 직렬화)과 글로벌 레인(전체 동시성 제한) 2단으로 줄 세우는 단계.

등장 ← 세션 레인 · 글로벌 레인 · 기록 압축 · 출구(17)

세션 레인 심화 step-02-session-lane

같은 대화를 정원 1로 직렬화하는 대화별 FIFO 줄. '정원이 정해진 줄'이라는 공용 기본기(레인)를 session:<대화> 이름으로 쓴 한 사례.

등장 ← 세션·큐(B) · 글로벌 레인 · 대화 주소·기록 파일 · 기록 압축 · 출구(17)

글로벌 레인 심화 step-02-global-lane

모든 대화가 함께 쓰는 공용 줄. 동시에 도는 '서로 다른 대화'의 수를 N(기본 4, agents.defaults.maxConcurrent)으로 제한한다 — 한 대화 안의 병렬이 아니다. 제공사 속도 제한·자원 폭주를 막는 안전밸브.

등장 ← 세션·큐(B) · 세션 레인

대화 주소 · 기록 파일 심화 step-02-session-address

한 대화를 가리키는 두 이름 — 논리 이름표 대화 주소(agent:<id>:<채널>:<상대>)와 실제 기록 파일(<id>.jsonl). 둘은 색인 sessions.json으로 이어지고, 본문은 사용자 기기 로컬의 덧붙이기 전용 JSONL로 산다(원격 DB 아님).

등장 ← 세션·큐(B) · 기록 압축

기록 압축 · 컴팩션 심화 step-02-compaction

대화가 길어져 모델 한도를 넘기 전에, OpenClaw가 옛 기록을 짧은 요약으로 줄이는 일. 줄일 시점·경로·디스크 기록은 OpenClaw가 조율하고 '요약' 한 토막만 모델에게 시킨다. 급하면 본선에서 즉시, 아니면 백그라운드로 미뤄 다음 턴 전 배리어로 마무리.

등장 ← 세션·큐(B)

C · 해석

모델·하네스·인증 해석 step-03-resolution

이번 턴을 무엇으로 돌릴지(추론 엔진·모델·계정) 확정하고, 막혔을 때 갈아끼울 채비까지 루프 전에 한 번 갖추는 단계.

등장 ← 어템프트 루프(D) · 시스템 프롬프트(11) · 에이전트 세션(13) · 스트림(14) · 복구(16) · 출구(17)

하네스 step-03-resolution

한 번의 시도를 실제로 굴리는 교체 가능한 추론 엔진 어댑터. OpenClaw 자체 엔진과 Codex·Copilot이 같은 슬롯에 꽂힌다. (계약은 출구에서 더 깊이 다룸.)

등장 ← 시스템 프롬프트(11) · 에이전트 세션(13) · 출구(17)

인증 프로필 심화 step-03-auth-profile

한 프로바이더에 로그인하는 계정/자격 하나(API 키·토큰·OAuth). 계정들을 건강한 순서로 줄 세우고(같은 종류는 라운드로빈), 막히면 다음으로 회전하며, 막힌 계정엔 사유별 지수 백오프 쿨다운을 매긴다. OAuth는 만료 전 자동 갱신.

등장 ← 해석(C) · 어템프트 루프(D) · 스트림(14) · 복구(16)

D · 어템프트 루프 · 조립 10단계

어템프트 루프 step-04-attempt-loop

한 턴을 여러 번 시도할 수 있는 재시도·페일오버 드라이버. 매 반복마다 이너 어템프트 1회를 디스패치하고 결과를 분류해 재시도/회전/폴백/컴팩션/종료를 정한다.

등장 ← 해석(C)

워크스페이스 · 샌드박스 step-05-workspace-sandbox

에이전트가 파일을 읽고 명령을 실행할 작업 공간 경로와, 격리 실행을 위한 샌드박스 컨텍스트를 해석한다.

등장 ← 스킬(6) · 도구(7) · 코드 모드(9) · 시스템 프롬프트(11)

스킬 step-06-skills

이 환경에서 이 에이전트가 실제로 쓸 수 있는 스킬만 추려, 목차를 스냅샷으로 갈무리해 매 턴 재사용한다.

등장 ← 설정 스냅샷 · 워크스페이스(5) · 도구(7) · 시스템 프롬프트(11) · 리소스 로더(12) · 스트림(14)

코어 · 플러그인 도구 step-07-core-plugin-tools

코어 도구와 플러그인 도구를 빌드해 합치고, allow-list로 이번 턴에 노출할 도구를 거른다.

등장 ← 번들 도구(10)

부트스트랩 컨텍스트 step-08-bootstrap-context

BOOTSTRAP·AGENTS 같은 컨텍스트 파일을 한 턴 조립에 주입해, 프로젝트 약속·지침을 모델 앞에 깔아 준다.

등장 — 본문 역링크 없음

코드 모드 · 도구 검색 step-09-code-mode-tool-search

도구를 한꺼번에 노출하는 대신 검색해 불러오는 카탈로그 방식을 (조건부로) 적용해, 거대한 도구 목록의 부담을 던다.

등장 — 본문 역링크 없음

번들 도구 step-10-bundle-tools

외부 MCP·LSP 도구 서버를 이번 턴의 도구 목록에 묶어, 모델이 외부 능력을 도구로 부를 수 있게 한다.

등장 ← 도구(7) · 코드 모드(9) · 출구(17)

시스템 프롬프트 step-11-system-prompt

시스템 프롬프트 쌍(base + provider 변환)을 빌드한다. 공통 골격을 짓고 프로바이더에 맞게 한 번 더 변환해 모델을 존중한다.

등장 ← 에이전트 세션(13) · 디스패치(15)

세션 리소스 로더 step-12-session-resource-loader

SessionManager와 컨텍스트 엔진을 부트스트랩해, 대화 기록·자원을 이번 턴이 쓸 수 있게 준비한다.

등장 — 본문 역링크 없음

에이전트 세션 step-13-agent-session

agent-core 세션을 생성하고 프롬프트를 설치해, 실제 한 턴을 굴릴 정체를 세운다.

등장 ← 리소스 로더(12)

스트림 셋업 step-14-stream-setup

실제 streamFn을 해석하고 transport를 적용해, 모델로 가는 실제 스트림 통로를 연다.

등장 ← 디스패치(15)

디스패치 · 복구 · 출구

디스패치 step-15-dispatch

LLM 스트림을 여러 겹으로 감싼 뒤 모델을 실제로 호출하고, 내부 도구 루프를 돌려 stop-reason까지 받아 온다.

등장 ← 어템프트 루프(D) · 스트림(14) · 출구(17)

모델↔도구 루프 심화 step-15-tool-loop

한 번의 모델 호출이 아니라, 모델이 도구를 부르고 그 결과를 되먹이며 한 턴 안에서 여러 바퀴 도는 되돌이. 답하기 전에 여러 번 행동·관찰하는 이 안쪽 루프가 챗봇과 에이전트를 가른다.

등장 ← 디스패치(15)

방어막 겹겹 (래핑) 심화 step-15-wrapping

모델로 가는 호출을 날것으로 보내지 않고 세션 호출 함수(streamFn) 위에 약 17겹(6겹은 무조건)을 둘러, thinking 재생 거부·도구 이름 공백 같은 프로바이더 결함을 사용자 모르게 흡수하는 경계 엔지니어링.

등장 ← 디스패치(15)

stop-reason 심화 step-15-stop-reason

모델↔도구 루프가 멎은 이유(완료·도구 호출·거부·길이 한계·오류). 이 멈춤의 이유가 이번 턴을 종결할지, 갈아탈 신호를 바깥에 띄울지를 가른다.

등장 ← 디스패치(15)

복구 — 페일오버 · 컴팩션 step-16-recovery

결과를 분류해 살아남기: 실패별 페일오버 사다리, 선제 컴팩션(요약은 최후·요약마저 감사), 재생 안전성으로 긴 세션을 손실·중복 없이 잇는다.

등장 ← 해석(C) · 어템프트 루프(D) · 출구(17)

페일오버 (실패 모양·처방 사다리) 심화 step-16-failover-ladder

실패를 모양별로 분류해 가장 싼 처방부터 올라가는 사다리. 공식 결정은 5액션(continue_normal·rotate_profile·surface_error·fallback_model·return_error_payload)이며, 가벼운 회복은 안쪽 루프가, '모델 교체'만은 바깥 드라이버가 맡는다.

등장 ← 복구(16)

선제 컴팩션 심화 step-16-preemptive-compaction

맥락 창이 터진 뒤 수습하지 않고, 보내기 전에 미리 재서 가장 적게 잃도록 줄이는 일. 압력에 따라 4-루트(fits · 도구출력 절단 · 요약+절단 · 요약만)로 갈리며, 첫 시스템·부트스트랩 맥락은 절대 가지치지 않는다.

등장 ← 복구(16)

요약 감사 심화 step-16-summary-audit

요약이 불가피할 때조차 모델이 만든 요약을 그대로 믿지 않고, 필수 다섯 섹션·식별자 보존·최신 요청 반영을 감사해 미달이면 안전한 대체본(이전 요약 또는 골격)으로 떨어지는 안전장치.

등장 ← 복구(16)

재생 안전성 심화 step-16-replay-safety

재시도 전에 '다시 돌려도 안전한가'를 부수효과 증거로 판정하는 규율. 메시지 전송·세션 spawn·cron 기록 등 흔적이 하나라도 있으면 재생을 막고, 증명할 수 없으면 보수적으로 막는다(fail-closed).

등장 ← 복구(16)

출구 — 종료 판정 step-17-exit

incomplete-turn을 분류해 terminal outcome을 정하고 결과를 반환한 뒤, finally 정리와 레인 unwind까지 마친다. 교체 가능한 하네스 계약도 여기서 더 깊이 다룬다.

등장 ← 어템프트 루프(D)
