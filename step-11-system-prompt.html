system-prompt

시스템 프롬프트를 만든다. 핵심은 덮어쓰지 않고 변환한다는 것 — base + provider 변환의 2단 구조.

무엇을 하나

  • 런타임 파라미터를 빌드(buildSystemPromptParams).
  • 시스템 프롬프트 pair를 빌드(buildAttemptSystemPrompt) + 리포트 생성.
  • 결과: { baseSystemPrompt, systemPrompt } — base는 보존, provider용은 변환본.

2단 변환 구조

  • 1단(base): buildEmbeddedSystemPrompt(...) — OpenClaw 소유 베이스 프롬프트.
  • 2단(provider 변환): transformProviderSystemPrompt({ context: { systemPrompt: baseSystemPrompt } }) — 프로바이더 훅이 base를 입력으로 받아 모델 특성에 맞게 변환본을 반환.
  • raw 실행 존중: isRawModelRun이면 provider 프롬프트를 빈 문자열로 보내 모델 기본 동작을 가리지 않는다. base는 진단·프롬프트 캐시 경계용으로 보존.

코드 위치

  • attempt.ts:2089 prepStages.mark (시스템 프롬프트 완료 지점)
  • attempt.ts:1919 buildSystemPromptParams · attempt.ts:1994 buildAttemptSystemPrompt
  • attempt-system-prompt.ts:39–59 2단 변환 본체 ({ baseSystemPrompt, systemPrompt } pair 반환)

왜 이게 품질을 지키나

어설픈 래퍼의 전형적 실패는 모델/프로바이더 무관하게 프롬프트를 강제 덮어쓰는 것이다. OpenClaw는 베이스를 만들되 프로바이더 변환 훅을 거치고, raw 실행은 아예 비워 모델을 존중한다. 모델의 사고를 가리지 않으면서 자기 운영 맥락만 얹는 첫 번째 구체적 장치다.
