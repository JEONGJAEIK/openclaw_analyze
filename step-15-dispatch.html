디스패치 — 모델 호출 + 도구 루프

조립이 끝난 뒤 실제로 모델을 부르는 곳. 모델 호출을 날것으로 던지지 않고 여러 겹 방어막으로 감싼 뒤, 예산을 점검하고 모델 왕복과 내부 도구 루프를 돌린다.

무엇을 하나 (순서대로)

  • ① LLM 스트림 다겹 래핑 — base streamFn 위에 캐시·thinking 정리·Anthropic 복구·tool-call 정규화/복구·idle-timeout·diagnostic 등을 겹겹이 씌운다.
  • ② 컨텍스트 조립 — 최종 메시지 윈도를 조립(assembleAttemptContextEngine).
  • ③ pre-prompt 예산 검사 — 보내기 직전 컴팩션 결정(shouldPreemptivelyCompactBeforePrompt). 자세히는 복구.
  • ④ 런타임컨텍스트 주입 + 디스패치 — 프롬프트/런타임컨텍스트 변환을 설치하고 activeSession.prompt()로 모델 왕복 + 내부 도구 호출 루프를 돈다.
  • ⑤ stop-reason 처리 — lastAssistant.stopReason을 읽어 결과를 만들거나 페일오버 결정을 끌어낸다.

코드 위치

  • attempt.ts:2952–3160 스트림 래핑 스택 (base resolveEmbeddedAgentStreamFn :2856 위에)
  • attempt.ts:3299–3352 컨텍스트 조립
  • attempt.ts:4599–4601 shouldPreemptivelyCompactBeforePrompt (pre-prompt 예산)
  • attempt.ts:4764–4804 런타임컨텍스트 주입 + activeSession.prompt()
  • attempt.ts:5532 · 5556 stop-reason 처리

LLM 경계 — 모델 호출을 감싸는 방어막

이 단계의 하이라이트. 세션의 streamFn(실제 모델 호출 함수)을 조건부로 겹겹이 래핑한다 — base 위에 약 17겹이 쌓이고, 그중 6겹은 무조건 적용된다(yield-abort·malformed 정리·text 승격·이름 trim·민감 stop-reason·diagnostic). 날것의 호출이 아니라 프로바이더 결함을 흡수하는 막을 두른 뒤 모델로 간다.

대표적인 겹:

  • 캐시 관측 — 프롬프트 캐시 적중/무효를 단계별 기록(cacheTrace.wrapStreamFn, attempt.ts:2958).
  • 메시지 정규화 — 과거 메시지에서 thinking/reasoning 블록 제거, 히스토리 오염 방지(attempt.ts:2966).
  • Anthropic 복구 — Claude 엔드포인트가 재생된 thinking 블록을 거부하면, thinking 없이 1회 재시도하고 성공 시 트랜스크립트를 자동 수리(attempt.ts:2983).
  • 도구호출 정규화/복구 — 망가진 tool-call을 파서 수준에서 흡수 → 복구.

왜 이게 품질을 지키나

날것 래퍼라면 프로바이더 결함 하나에 턴이 깨진다. 실제 attempt.ts 주석이 그 사례를 말한다:

"Anthropic Claude endpoints can reject replayed thinking blocks on any follow-up provider call… Sanitize outbound messages where policy allows rewriting; otherwise preserve latest thinking and let the recovery wrapper retry once without it."

OpenClaw는 정책 기반 정규화 + 자동 재시도 + 트랜스크립트 수리로 사용자가 모르게 넘어간다. 비법은 프롬프트 마법이 아니라, 프로바이더 결함을 흡수하는 경계 엔지니어링이다.
